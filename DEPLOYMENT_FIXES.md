# Deployment Fixes Applied

## Issues Fixed

### 1. **Missing curl in Dockerfile**
- **Problem**: Healthcheck was failing because `curl` wasn't installed in the runtime container
- **Fix**: Added `RUN apt-get update && apt-get install -y curl` to the backend Dockerfile
- **Frontend**: Updated Dockerfile to use `curl` for consistency and added `apk add --no-cache curl`

### 2. **Environment Variable Mismatch**
- **Problem**: Application expected `SPRING_DATASOURCE_URL`, `APP_JWT_SECRET` but Render was providing `DATABASE_URL`, `JWT_SECRET`
- **Fix**: Updated `application-prod.properties` to map the correct environment variable names:
  - `DATABASE_URL` → `spring.datasource.url`
  - `DB_USER` → `spring.datasource.username`
  - `DB_PASSWORD` → `spring.datasource.password`
  - `JWT_SECRET` → `app.jwt.secret`
  - `ALLOWED_ORIGINS` → `app.cors.allowed-origins`

### 3. **Missing Defaults**
- **Problem**: Required environment variables had no defaults, causing startup failures
- **Fix**: Added sensible defaults to all configuration properties for local development compatibility

### 4. **Dockerfile Path Issues**
- **Problem**: `render.yaml` had incorrect Dockerfile paths (`./Dockerfile` and `./cafe-backend`)
- **Fix**: Updated paths to:
  - Backend: `./cafe-backend/Dockerfile` with `dockerContext: ./`
  - Frontend: `./cafe-frontend/Dockerfile` with `dockerContext: ./`

### 5. **Frontend API URL**
- **Problem**: Frontend environment variable wasn't including `/api` path
- **Fix**: Updated `VITE_API_URL` value to `https://cafe-backend.onrender.com/api`

### 6. **Actuator Configuration**
- **Problem**: Health endpoint wasn't properly exposed
- **Fix**: Added actuator configuration in `application-prod.properties`:
  ```properties
  management.endpoints.web.exposure.include=health
  management.endpoint.health.show-details=when-authorized
  ```

### 7. **Production Security**
- **Problem**: Swagger UI was enabled in production
- **Fix**: Disabled Swagger UI in production mode for security

## Updated Files

1. **cafe-backend/Dockerfile**
   - Added curl installation
   - Increased health check start-period to 60s for database connection

2. **cafe-backend/src/main/resources/application-prod.properties**
   - Fixed environment variable names
   - Added defaults for all properties
   - Disabled Swagger UI
   - Added actuator endpoints configuration

3. **render.yaml**
   - Fixed Dockerfile paths
   - Corrected environment variable bindings
   - Added `/api` to frontend API URL

4. **cafe-frontend/Dockerfile**
   - Added curl for health checks
   - Updated healthcheck timeout and start-period

## Deployment Checklist

Before pushing to production:

1. ✅ Update `repo` URLs in `render.yaml` with your GitHub repository URLs
2. ✅ Verify database connection string format in Render PostgreSQL service
3. ✅ Check that all environment variables are properly set
4. ✅ Ensure `.gitignore` is configured correctly (already done)
5. ✅ Test locally with `docker-compose` before deploying

## Environment Variables Reference

### Backend (cafe-backend)
- `DATABASE_URL`: PostgreSQL connection string (provided by Render)
- `DB_USER`: Database username (provided by Render)
- `DB_PASSWORD`: Database password (provided by Render)
- `JWT_SECRET`: JWT signing key (auto-generated by Render)
- `ALLOWED_ORIGINS`: Comma-separated list of allowed origins for CORS
- `SPRING_PROFILES_ACTIVE`: Set to `prod` (automatic)

### Frontend (cafe-frontend)
- `VITE_API_URL`: Backend API URL (e.g., `https://cafe-backend.onrender.com/api`)
- `NODE_ENV`: Set to `production` (automatic)

## Troubleshooting

If deployment still fails:

1. **Check logs**: View Render logs for specific error messages
2. **Database connection**: Verify PostgreSQL service is running and accessible
3. **Health check**: Test the health endpoint directly
4. **Environment variables**: Confirm all variables are properly set in Render dashboard

See `DEPLOYMENT_GUIDE.md` for more detailed information.
